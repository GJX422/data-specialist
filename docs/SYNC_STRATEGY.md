# 🔄 同步策略详解

## 问题背景

传统的数据同步方式存在严重遗漏问题：

### 传统方式：`--days N`

```bash
# 只获取最近N天新建的记录
python3 sync_script.py --days 30
```

**问题**：
- 审批中的老记录会被遗漏（可能持续数周甚至数月）
- 待提交的草稿记录无法同步
- 导致报表数据不完整

### 实际案例

以费用报销为例：
- 传统方式（--days 30）：只能同步最近30天新建的报销单
- 实际情况：有127条审批中的报销单，最早创建于数月前
- 结果：报表严重低估费用支出

## 智能同步策略

### 核心思路

对状态类表采用**两趟查询**策略：

1. **第1趟**：获取近30天新建记录（覆盖近期业务）
2. **第2趟**：获取所有审批中/待提交记录（不遗漏任何活跃记录）

### 实现方式

#### 编排器层面

```python
# yunzhijia_sync_all_mysql.py

# 状态类表：执行两趟查询
for script in STATUS_BASED_SCRIPTS:
    # 趟1：近30天新建
    script.run(start=30_days_ago, end=today)
    
    # 趟2：所有审批中/待提交
    script.run(status="RUNNING,RETURNED")
```

#### 单脚本层面

```python
# yunzhijia_sync_expense_reimbursement_mysql.py

if __name__ == "__main__":
    parser = argparse.ArgumentParser()
    parser.add_argument("--start", help="开始日期")
    parser.add_argument("--end", help="结束日期")
    parser.add_argument("--status", help="状态过滤，逗号分隔")
    parser.add_argument("--all", action="store_true", help="全量同步")
    
    args = parser.parse_args()
    
    if args.status:
        # 按状态过滤
        sync_by_status(args.status.split(","))
    elif args.start and args.end:
        # 按日期范围
        sync_by_date_range(args.start, args.end)
    elif args.all:
        # 全量同步
        sync_all()
```

### 状态值映射

云之家API使用`rawValue`作为状态过滤值：

| 显示值 | rawValue | 说明 |
|--------|----------|------|
| 审批中 | `RUNNING` | 正在审批流程中 |
| 已完成 | `FINISH` | 审批通过并完成 |
| 待提交 | `RETURNED` | 被退回，待重新提交 |
| 已废弃 | `ABANDON` | 已取消或废弃 |

**重要**：必须使用`rawValue`，使用显示值（如"审批中"）会导致查询返回0条。

## 产线同步策略

### 独立全量同步

7条产线均有独立的全量同步cron任务，每天执行4轮：

| 产线 | Cron时间 | 说明 |
|------|----------|------|
| 鲜品采收 | 02:35/09:05/13:05/18:05 | 明细表全量同步 |
| 爱尔康生产 | 02:37/09:07/13:07/18:07 | 生产明细表全量同步 |
| 喷雾干燥 | 02:39/09:09/13:09/18:09 | 生产明细表全量同步 |
| 闪蒸干燥粉 | 02:41/09:11/13:11/18:11 | 产出明细全量同步 |
| 成品破壁粉 | 02:43/09:13/13:13/18:13 | 产出明细全量同步 |
| 饲料原料 | 02:45/09:15/13:15/18:15 | 产出明细全量同步 |
| 终端品 | 02:47/09:17/13:17/18:17 | 销售订单全量同步 |

### 日期字段选择

**重要**：生产记录常为补录，`制单日期` ≠ `生产日期`。

| 产线 | 正确日期字段 | 错误日期字段 |
|------|-------------|-------------|
| 鲜品采收 | `鲜品采收明细.日期` | 记录表.制单日期 |
| 爱尔康生产 | `爱尔康生产明细.生产日期` | 记录表.制单日期 |
| 喷雾干燥 | `喷雾干燥生产明细.生产日期` | 记录表.制单日期 |

查询生产产出必须用`生产日期`，用`制单日期`会导致补录数据被误认为"当日产出"。

## 数据完整性保障

### 同步后验证

每次同步后需验证：
1. 记录数是否符合预期
2. 关键字段是否有空值
3. 日期范围是否正确

### 数据健康看板

通过数据健康看板实时监控：
- 🟢 绿灯：数据正常
- 🔴 红灯：数据异常/缺失
- 🟡 黄灯：数据待确认

部署地址：https://www.htmlcode.fun/s/data-health-board

## 性能优化

### 批量upsert

使用MySQL的`INSERT ... ON DUPLICATE KEY UPDATE`实现批量upsert，避免重复插入。

### 分页查询

API查询采用分页机制（每页100条），避免单次请求数据量过大。

### 并发控制

- 产线同步：7条产线串行执行，避免API限流
- 状态类同步：9个脚本串行执行，确保数据一致性
